project

context

what i am building

a data cleaning toolkit that keeps cleaning decisions visible. the aim is not just to output a cleaner file, but to make the work easier to justify afterwards.

  • simple web interface
  • open source implementation
  • repeatable cleaning behaviour

why it matters

cleaning can change which rows survive, which values count as missing, and what a later model learns. if those steps are hidden, the result is harder to trust.

  • clearer decisions
  • easier review
  • less black-box cleanup
design

tool approach

privacy-first handling

files are sent over https, processed in memory, and discarded after cleaning. they are not written to disk, logged, or shared with third parties.

  • in-memory processing
  • no file storage
  • no account required

technical direction

the core algorithms are written in c++ and exposed through simple api endpoints. browser-based processing through webassembly is the next major direction.

  • deterministic algorithms where possible
  • small, inspectable cleaning steps
  • open implementation
research

evaluation

how it is tested

the toolkit is evaluated with benchmark datasets that contain known quality issues, then compared against existing cleaning workflows and scripts.

  • before/after counts
  • timing measurements
  • workflow comparison

fairness angle

cleaning can shift data distributions. the project treats transparency as a safety feature because visible cleaning steps are easier to question and correct.

  • distribution changes matter
  • rules should be explainable
  • outputs should be auditable
academic

details

university context

this is a final-year honours project at edinburgh napier university, supervised by dr. lynn langit.

the dissertation examines whether explicit, repeatable cleaning steps can improve auditability in machine learning workflows.
